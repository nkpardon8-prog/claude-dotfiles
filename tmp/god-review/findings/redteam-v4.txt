### [catastrophic] `exit 0` inside per-finding bash fences will exit the entire harness shell, terminating /god-review mid-loop
**Source:** redteam
**Evidence:** god-review.md:1148, 1156, 1178, 1201, 1207 — sub-step 3e validation fences use `exit 0` to "continue to next AUTO_FIX finding." Comments next to each say "# Continue to next AUTO_FIX finding."
**Impact:** A Bash tool call invocation in Claude Code runs in a shell session that persists state via the harness wrapper, but `exit 0` in the body terminates THAT bash invocation cleanly — and crucially aborts the rest of the inline orchestration in that fence. More importantly, the orchestrator semantics here are "skip this finding, advance to the next one," but `exit 0` at the orchestrator-prose level means the orchestrator (LLM) sees a clean exit and may interpret it as "this entire sub-step succeeded" rather than "this one finding was skipped." Per the Phase G design, control flow is supposed to live in orchestrator reasoning, NOT in bash exit codes. Using `exit 0` to mean "continue per-finding loop" conflates "bash block ended successfully" with "skip-and-continue" — the LLM has no way to distinguish a malformed-Architect skip (line 1148) from a clean validation success at the end of a normal pipeline. This was supposed to be fixed in Phase G but the `exit 0` pattern survived from the original bash-while-loop design.
**Fix:** Replace each `exit 0` with `echo "SKIP_TO_NEXT_FINDING: <reason>"; SKIP_THIS_FINDING=true` and have the orchestrator-prose check `$SKIP_THIS_FINDING` before each subsequent sub-step (iv, v, vi, vii). Or: split each sub-step into its own bash fence and let the orchestrator skip subsequent fences for skipped findings via prose-level reasoning.

### [catastrophic] Architect's prompt instructs it to write JSON to disk but Editor's prompt assumes Read tool can read absolute paths the harness has never granted
**Source:** redteam
**Evidence:** god-review.md:1098-1131 (Architect spawn) writes to `$WORKDIR/tmp/god-review/architect-output-${FINDING_ID}.json` — the path is built from `$WORKDIR` which can be an arbitrary repo. god-review.md:1218-1222 (Editor spawn) tells the Editor to "Read it via the Read tool and apply the change exactly: $ARCH_OUTPUT_FILE". But the Editor sub-agent has no guarantee its Read tool is permitted on that absolute path — agents inherit the orchestrator's tool permission scope, but the agent harness routinely sandboxes sub-agents to the cwd they're spawned in.
**Impact:** If the Editor sub-agent is harness-sandboxed (or simply confused by an absolute path it didn't construct), Read returns "permission denied" or "file not found." Editor returns EDITOR_ABORT, the fix gets reverted, the finding's hash gets added to `finding_history_hashes` (line 1249) — meaning it's now permanently in the replay-skip list. Every fix attempt in /god-review fails this way, and the loop runs to wall-clock cap producing zero kept fixes while marking every finding as "tried-and-reverted."
**Fix:** Either (a) pass the JSON contents inline to the Editor prompt (defeats the G2 disk-based-output fix's purpose), or (b) explicitly tell the Editor in its prompt "the Architect's JSON is at this path — Read tool is permitted; if Read fails, treat as EDITOR_ABORT with reason 'cannot-read-architect-json'", and augment editor-agent.md with a fallback to read via Bash `cat` if Read tool fails. Better: have the orchestrator read the file itself and pass the parsed JSON fields as separate prompt placeholders to Editor.

### [high] sub-step 3g writes round number to state.json but never increments BEFORE the per-round triage; round 1 records as round 0
**Source:** redteam
**Evidence:** god-review.md:865-867 — `ROUND=1` set on first entry. Lines 1546-1547 — `d['round'] = $ROUND` written at end of round 3g. But line 1567 increments `ROUND=$((ROUND + 1))` only AFTER the state.json write at 1542-1553. So state.json's `round` field reflects the round just completed, not the round about to start. Then on `--resume` (line 862), the orchestrator reads `round` from state.json and starts there — which means a resumed run repeats the round that was just completed (because we record N at end of round N, then read N at start of resumed round, then re-do N).
**Impact:** Off-by-one on resume. Findings from "the last completed round" are re-processed on resume; their hashes are already in `finding_history_hashes` (for reverted ones) but NOT for kept ones (per line 1382: KEEP path doesn't record hash). So a kept fix gets re-attempted on resume — applying the same fix twice can produce duplicate edits or "already-applied" pre-commit failures, both of which path through to revert and corrupt the audit trail.
**Fix:** On resume, `ROUND = state.round + 1`. Or: write state.json's round AFTER incrementing in 3g.
