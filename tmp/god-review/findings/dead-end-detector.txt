## F1 [definite, CRITICAL] frozen_added_per_round + architect_malformed_per_round — read but never written

Location: god-review.md:1330-1331 (read sites in AVG_INSTABILITY computation)
Evidence: grep -rFn "frozen_added_per_round" returns ONE hit (the read at 1330). grep -rFn "architect_malformed_per_round" returns ONE hit (the read at 1331). Neither field is in the state.json initializer (lines 384-401) and no python block writes them. The churn detector (lines 1093-1116) appends to `frozen_units` but never to `frozen_added_per_round`. The malformed-architect handler (lines 938-947) increments `architect_malformed_count` but never `architect_malformed_per_round`.
Impact: AVG_INSTABILITY is permanently 0. The --loop instability abort (line 1337, "if AVG > INSTABILITY_RATE") is dead code — the loop can never terminate via this path. INSTABILITY_RATE constant (line 166) becomes a partial dead-end too because its consumer is never fed real data.
Fix: Either (a) write to these fields each round (push len(frozen_units_added) and architect_malformed_count delta into the per-round arrays), or (b) read from existing fields (frozen_units length delta, architect_malformed_count delta).

## F3 [definite, HIGH] SPINLOCK_TIMEOUT_SEC — Tunable Constant orphaned from its consumer

Location: god-review.md:171 (declaration); codex-invoke.sh:98 (hardcoded duplicate)
Evidence: `SPINLOCK_TIMEOUT_SEC="${SPINLOCK_TIMEOUT_SEC:-600}"` declared in orchestrator. codex-invoke.sh:98 hardcodes `LOCK_TIMEOUT=600` instead of reading SPINLOCK_TIMEOUT_SEC. codex-invoke.sh is run via `bash` (not source) and does not export the constant from the orchestrator's environment in any case.
Impact: Override via env var has no effect on actual spinlock; the documented tunable is a lie.
Fix: In codex-invoke.sh, replace the hardcoded `LOCK_TIMEOUT=600` with `LOCK_TIMEOUT="${SPINLOCK_TIMEOUT_SEC:-600}"` — and ensure the orchestrator exports SPINLOCK_TIMEOUT_SEC before invoking the script.

## F4 [definite, HIGH] FROZEN_UNITS_CAP — self-referencing assignment with no default

Location: god-review.md:167
Evidence: `FROZEN_UNITS_CAP="$FROZEN_UNITS_CAP"` (note the missing `:-N` fallback, unlike every other Tunable Constant on lines 164-171). If the env var is unset, FROZEN_UNITS_CAP is the empty string. Then line 1363 `[ "$FROZEN_UNITS_COUNT" -gt "$FROZEN_UNITS_CAP" ]` fails with "integer expression expected" and the bounded-mode escalation never triggers.
Impact: The frozen-units escalation gate is silently broken in default usage. A user not setting FROZEN_UNITS_CAP gets no protection.
Fix: `FROZEN_UNITS_CAP="${FROZEN_UNITS_CAP:-3}"` (or chosen default).

## F5 [definite, HIGH] MAX_ROUNDS_EXPLICIT — flag parsed, dies at bash-fence boundary

Location: god-review.md:36, 57, 1309 (parser sets it; termination check reads it; intervening write_env does NOT persist it)
Evidence: Parsed at line 57 (`MAX_ROUNDS_EXPLICIT=true`). env-helpers.sh:20-24 (write_env's persisted-var list) does NOT include MAX_ROUNDS_EXPLICIT. Many bash blocks between Step 0 (parsing) and Phase 3 (line 1309) source .env.sh and run write_env, so by the time the termination check reads `$MAX_ROUNDS_EXPLICIT` it is empty (sourced from .env.sh which doesn't define it).
Impact: B12 ceiling check (`if [ "$MAX_ROUNDS_EXPLICIT" = "true" ] && [ "$ROUND" -ge "${MAX_ROUNDS:-9999}" ]`) always evaluates false. The user-facing guarantee that --max-rounds caps --loop is broken.
Fix: Add MAX_ROUNDS_EXPLICIT to env-helpers.sh write_env var list.
