# gap-detector v3 — Phase E v3 audit of /god-review + /god-report

### [catastrophic] Phase 3 round-loop has no actual loop construct — orchestrator-prose says "re-enter 3a" with no mechanism
**Source:** gap-detector
**Evidence:** god-review.md:867-873, 1449-1473 — prose says "After each round completes, the round-end decision (sub-step 3g) tells you (the orchestrator) whether to re-enter at 3a... Execute these sub-steps in order each round." Then 1449: "Now you (the orchestrator) make the loop-back decision based on `NEW_NEW_FINDINGS`". There is no machine-readable jump target, no harness loop construct, no `goto`, no slash-command-self-invocation. The orchestrator (LLM) is asked to "re-enter Phase 3 sub-step 3a directly" purely via prose. The single bash fences in 3a–3g each exit normally; nothing causes the next iteration to actually happen.
**Impact:** The "always-on indefinite fix loop" advertised in the frontmatter is not implementable as written. An LLM running the spec will execute round 1, hit 3g, and then have no concrete instruction to physically re-emit sub-step 3a tool calls — there is no enclosing while/for/recursion. In practice the model will probably stop at round 1 or 2, defeating the entire termination criterion ("3 consecutive clean rounds").
**Fix:** Either (a) wrap the entire 3a-3g sequence in an explicit orchestrator-prose loop counter with a clearly stated rule "if not terminated, you MUST re-issue the 3a bash block as your next assistant message, even if context is large", or (b) add a single canonical bash while-loop scaffold around 3a-3g (one fence containing the round body, with the Agent calls described in prose inside).

### [catastrophic] Architect→ARCH_OUTPUT capture has no defined capture mechanism
**Source:** gap-detector
**Evidence:** god-review.md:1065-1067 — "Capture the agent's result text into `ARCH_OUTPUT` (a string variable in your reasoning — when you run the next bash block, you substitute the captured text into `ARCH_OUTPUT='...'` literally)." Then 1075: `ARCH_OUTPUT='<paste the architect output JSON here, single-quoted>'`. The instruction tells the LLM to literally edit the bash fence text between two assistant messages.
**Impact:** This is the load-bearing data hand-off between the Architect agent and the validation/Editor stages. Asking the orchestrator-LLM to inject Agent tool result text *as a literal Python-quoted string into a placeholder inside a bash fence* is fragile and has no schema for handling: result text containing single quotes, multi-line strings, ANSI color codes, the agent's own preamble before the JSON. There is no `cat > tmp/god-review/arch-output.json` instruction telling the agent to write its JSON to disk. JSON parse errors downstream (line 1097) will fire on every Architect output that contains apostrophes in `"rationale"`.
**Fix:** Instruct the Architect agent to write its JSON to `tmp/god-review/arch-output-${FINDING_ID}.json` via Bash heredoc, and have the validation block read from that file (`ARCH_OUTPUT=$(cat tmp/god-review/arch-output-${FINDING_ID}.json)`). Eliminate the "paste literal text" hand-off.

### [catastrophic] Sub-step 3f references undefined `$PRE_FIX_BASE_REF`
**Source:** gap-detector
**Evidence:** god-review.md:1343 — verifier prompt template includes `$(git diff $PRE_FIX_BASE_REF..HEAD)`. `$PRE_FIX_BASE_REF` is never defined anywhere in the file (grep -n confirms 1 hit, the reference itself). The 3e snapshot uses `PRE_FIX_REF` (line 1024). Round-start snapshot uses `REF` (line 365).
**Impact:** When verifier-1 prompt is rendered, `$PRE_FIX_BASE_REF` expands to empty string and `git diff ..HEAD` fails or emits no diff. Verifiers either get no diff context (silently re-review whole tree) or the bash interpolation fails outright. Either way, the verifier sub-pass is broken and `VERIFIER_NEW` is unreliable — and that count drives termination at 3g.
**Fix:** Either rename to `PRE_FIX_REF` (per-finding pre-fix snapshot from 3e), or define `PRE_ROUND_REF` at the top of each round capturing `git rev-parse HEAD` before any 3e fixes ran. Document which is intended; the round-level diff is more useful for verifiers since per-finding snapshots are stomped on each successive 3e iteration.

### [high] Sub-step 3f verifier instructions don't specify the prompts concretely enough to spawn
**Source:** gap-detector
**Evidence:** god-review.md:1338-1349 — "Verifier 2: same shape but model `claude-opus-4-7` with different focus prompt (correctness vs. architecture)" and "Verifier 4: another Codex agent, different prompt focus." No prompt template, no principle file path, no broad-reviewer file path. The "different focus" is undefined.
**Impact:** This is the verifier sub-pass that the catastrophic Phase E finding from v1 specifically demanded be made concrete. The instruction "different prompt focus" is exactly the prose-without-specification anti-pattern the gap-detector exists to find. Two of the four verifiers cannot be spawned; the orchestrator will either skip them or improvise prompts — losing reproducibility.
**Fix:** Wire each verifier to a concrete file: e.g. Verifier-1 = `broad-reviewers/claude-deep-correctness.md`, Verifier-2 = `broad-reviewers/claude-architecture-prod.md`, Verifier-3 = `broad-reviewers/codex-cross-layer.md`, Verifier-4 = `broad-reviewers/codex-prod-scalability.md`. Or just say "spawn the same 4 broad reviewers from Phase 2 with diff context appended."

### [high] `MIRROR_MODE=dual` documented but never honored at any write site
**Source:** gap-detector
**Evidence:** god-review.md:119-122 — "If MIRROR_MODE = 'dual': write outputs to BOTH ~/.claude-dotfiles/ AND ~/.claude/ paths." `MIRROR_MODE` is set but never read again in the file (grep confirms no further reference). All write sites use `tmp/god-review/...` (relative to $WORKDIR, which is the repo, not either dotfiles tree) or `$HOME/.claude-dotfiles/...` (always single).
**Impact:** Documentation describes a dual-write feature that does not exist in the spec. If a user sets `.mirror-mode` to `dual`, nothing different happens. False assumption hazard — user thinks both trees stay in sync without the auto-sync hook.
**Fix:** Either implement: every write that targets dotfiles paths must check `[ "$MIRROR_MODE" = "dual" ]` and write to `~/.claude/...` mirror as well, or delete the documentation. Since `tmp/` writes go to the repo (not dotfiles), in practice MIRROR_MODE only affects `lib/known-deferred.txt` promotion at line 1547. Either delete the dual-write prose or wrap that write in the conditional.

### [high] `check_phase_drift` is a no-op for the current architecture but documented as drift detector
**Source:** gap-detector
**Evidence:** lib/env-helpers.sh:329-337 — `check_phase_drift` does `awk '/^## Phase 0/{p=1} p' "$b"` on god-report.md. But god-report.md does NOT contain Phase 0/1/2 bodies — it delegates ("Read the spec... Open `god-review.md` and execute exactly the contents of: `## Phase 0: Context Map`..."). So awk extracts nothing meaningful, diff always reports difference (or always empty depending on parsing). god-report.md:130-136 instructs the user to run `check_phase_drift` after edits.
**Impact:** The drift check is broken for the actual architecture chosen (delegation, not duplication). It will either always pass (vacuously) or always warn — neither is signal. Worse, README/docs imply this is a real safety net.
**Fix:** Either (a) delete `check_phase_drift` and the documentation invocation, or (b) reimplement: parse god-report.md's "delegated phase" markers and verify god-review.md still has those exact section headers. The current diff-on-extracted-text logic doesn't match the delegated-execution architecture.

### [high] ACTIVE_PRINCIPLES is pseudocode only — never becomes a real shell array used to drive Phase 2 spawns
**Source:** gap-detector
**Evidence:** god-review.md:483 defines `ACTIVE_PRINCIPLES = ALWAYS_ON_PRINCIPLES + [p for p,sig in STACK_GATED_PRINCIPLES if sig != ""]` inside a triple-backtick block with no language tag. It's Python-list-comp pseudocode in a fenced block that an LLM might or might not execute. Line 595: "For each principle in ACTIVE_PRINCIPLES, spawn one Agent tool call". The orchestrator must construct this list itself from prose; there is no bash block exporting `ACTIVE_PRINCIPLES=("single-pattern" "reuse" ...)` to drive the spawn loop.
**Impact:** Each round, the orchestrator-LLM must reconstruct the active-principle list from the HAS_* signals via reasoning. Drift risk: the LLM might forget one of the 4 new failure-class detectors (dead-end, info-loss, contradiction, gap) that were added in fix-pass v2 (per CLAUDE.local.md). This is exactly the failure mode caught by Phase E v1 finding "4 new principles never spawn."
**Fix:** Convert the pseudocode to a real bash block: build a bash array `ACTIVE_PRINCIPLES=()` populated from ALWAYS_ON + stack-gated checks, exported via `write_env`. Then Phase 2c's prose can say "for each name in $ACTIVE_PRINCIPLES, spawn..." and the orchestrator iterates a concrete list.

### [medium] `RATIONALE` referenced in commit message but never extracted from ARCH_OUTPUT

### [medium] `PRE_FIX_REF` set per-finding but `git checkout -- "$ARCH_FILE"` discards changes ignoring the snapshot
**Source:** gap-detector
**Evidence:** god-review.md:1024 captures `PRE_FIX_REF` (stash or commit). On EDITOR_ABORT (1147-1154), the spec uses `git stash apply $PRE_FIX_REF` for stash-type but `git checkout -- "$ARCH_FILE"` for commit-type. But on gate-fail (1229) and perf-fail (1272) and pre-commit-reject (1313), it's always `git checkout -- "$ARCH_FILE"` regardless of `PRE_FIX_REFTYPE`. The PRE_FIX_REF stash created at 1024 is never applied anywhere except in EDITOR_ABORT.
**Impact:** If the working tree was dirty before the fix (so `PRE_FIX_REF` is a stash), and gates fail post-Editor, `git checkout -- "$ARCH_FILE"` reverts only the Architect's targeted file. Any unrelated dirty state is preserved (correct — that's intended). But the stash created in step (i) is now an orphan — it lives in the stash tree forever, unreferenced. After many fix-cycles, `git stash list` accumulates stale `god-review pre-fix ...` entries.
**Fix:** Either: (a) drop the stash creation entirely on dirty trees (the targeted `git checkout -- "$ARCH_FILE"` is already sufficient since the Editor only touched one file), or (b) explicitly clean up unused pre-fix stashes at end of round 3g via `git stash drop` for refs not in `kept_fixes`.

### [low] State init in 1b uses `LOOPMODEPLACEHOLDER` substituted via sed, but bash `LOOP_MODE=true` produces lowercase `true`, while JSON requires unquoted `true`
**Source:** gap-detector
**Evidence:** god-review.md:374 `LOOP_MODE=true`, line 389 `"loop_mode": LOOPMODEPLACEHOLDER`, line 407 `sed -i.bak -e "s|LOOPMODEPLACEHOLDER|$LOOP_MODE|g"`. After sed, the resulting JSON line is `"loop_mode": true,` — that's actually valid JSON. OK on this one, but `MAXWALLPLACEHOLDER` is sed'd with `$MAX_WALL_HOURS_VAL` which defaults to `24` (numeric, fine). **Impact:** Brittle substitution — any future addition that's a string would need quoting. Document the contract.
**Fix:** Add a comment noting "placeholders that must remain JSON-valid: bools (`true`/`false`) and numbers only — strings must be wrapped in `\"...\"` in the heredoc."

### [low] `tmp/god-review/state.json` referenced as relative path in many places, but resolution depends on `cd` to repo root
**Source:** gap-detector
**Evidence:** Multiple bash blocks (e.g. line 87, 94, 99) reference `tmp/god-review/state.json` (relative). `WORKDIR` is set to repo root via `git rev-parse --show-toplevel`, but `cd "$WORKDIR"` is never executed at the start of bash blocks. ### [minor] `tmp/god-review/perf-baseline.json` written by 1c using `npm run bench 2>&1 > file` — both stderr and stdout merged
**Source:** gap-detector
**Evidence:** god-review.md:429-431 — `npm run bench 2>&1 > tmp/god-review/perf-baseline.json`. The 2>&1 before `>` redirects stderr to old stdout, then `>` redirects stdout to file — leaving stderr on terminal. Should be `> file 2>&1` to capture both. The downstream perf-regression parser at 1265 expects the file to contain bench results parseable as `(\w[\w\-]+).*?(\d+\.?\d*)\s*ms` regex — non-deterministic when stderr noise is missing.
**Impact:** Half-broken baseline capture. Perf regression detection may fire false positives or false negatives based on whether stderr or stdout had the timing numbers.
**Fix:** Change to `> tmp/god-review/perf-baseline.json 2>&1` consistently across all four bench script invocations (lines 429-435 + 1255-1258).

### [minor] `tmp/god-review/findings/` not cleared between rounds
**Source:** gap-detector
**Evidence:** Phase 2c `write_agent_finding` writes to `findings/<name>.txt`. If a round 2 broad-Claude reviewer is named the same as round 1 (which it always is, single name like "claude-broad-deep-correctness"), round 2 overwrites round 1. The verifier sub-pass (3f) at line 1353 uses `verifier-<n>` naming — same overwrite behavior.

**Fix:** Use `findings/round-${ROUND}/<name>.txt` for per-round subdirs. Phase 2d consolidation reads the latest round only; debugging can `ls findings/round-*/`.
