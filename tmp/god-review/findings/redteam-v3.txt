### [catastrophic] HAS_BENCH_SCRIPT python3 expression has unbalanced parens (regression of fix-pass v2 fix #64d5e98)
**Source:** redteam
**Evidence:** god-review.md:199 ‚Äî `python3 -c "import json; d=json.load(open('$WORKDIR/package.json')); scripts=d.get('scripts',{}); print(next((k for k in scripts if k in ('bench','benchmark','perf')),'')"` ‚Äî opens 4 parens in `print(next((k ... ),'')` but closes only 3. **Impact:** Any repo with `package.json` that lacks `bench`/`benchmark`/`perf` scripts will silently produce empty output via `2>/dev/null`. Any repo whose package.json HAS those scripts ALSO produces empty output because the script crashes before reaching the `print`. Stack-gating for perf-benchmark principle silently degrades to fall-through path (line 201 onward), which by chance finds bench/benchmark dirs, so the bug is masked unless you only have a package-json scripts-only signal. Fix-pass-v2 commit table verification command would NOT pass.
**Fix:** Wrap the inner generator: `print(next((k for k in scripts if k in ('bench','benchmark','perf')),''))` ‚Äî add the missing `)` before the closing `"`. Then re-run `python3 -c "<expr>"` to confirm.

### [high] Phase 3 stash-apply revert path leaks stashes (no `git stash drop`)
**Source:** redteam
**Evidence:** god-review.md:1151 ‚Äî `git stash apply "$PRE_FIX_REF" 2>/dev/null || true`. The PRE_FIX_REF is created by `git stash create` (line 1024) which produces a dangling stash object NOT in the stash list. After `apply`, the dangling object remains in `.git/objects` as garbage but more importantly, stash-apply is non-fatal-on-conflict ‚Äî if the working tree was modified between the stash-create and now (which is the entire purpose of the apply), conflicts produce `<<<<<<< HEAD` markers in the source files with no abort path. No `git checkout --` or `git reset --hard $PRE_FIX_REF` follows.
**Impact:** Per-finding revert in Phase 3e (Editor abort path) can produce conflict markers in real source files without notification. Subsequent gates (typecheck/lint) will fail on the conflict markers, the loop will mark this as a churn event, freeze the file, and never revert cleanly. Worse: combined with `--no-verify` prohibition, the resulting `git commit` will fail with "fix conflicts" error not understood by the orchestrator's pre-commit-rejected branch.
**Fix:** Replace `git stash apply` with a hard reset semantics for revert: if PRE_FIX_REFTYPE=stash, do `git checkout -- "$ARCH_FILE"` first (targeted), or `git checkout "$PRE_FIX_REF" -- "$ARCH_FILE"` to restore from the stash tree. Never use stash-apply in an unattended loop.

### [high] glob_to_regex output for non-slash patterns conflates "starts-anywhere" with "matches-anywhere-in-name"
**Source:** redteam
**Evidence:** env-helpers.sh:69-71 ‚Äî bare-name pattern `package.json` becomes `^(?:.*/)?package\.json$`. Self-test passes (`frontend/package.json` ‚Üí MATCH, `package.json` ‚Üí MATCH). ### [medium] glob_to_regex middle-`**` segment "/(?:.*/)?" ‚Äî risk of edge-case match for paths with literal "//"
**Source:** redteam
**Evidence:** env-helpers.sh:85 ‚Äî `out.append('/(?:.*/)?')`. Self-test cases pass for normal paths but doesn't cover `//` (double-slash) edge cases that appear in glob normalizers / OS-quirks. Also uncovered: pattern `**/X/**` produces `^(?:.*/)?/X/.*$` ‚Äî note the leading `/` after `(?:.*/)?` since the middle pattern emits `/(?:.*/)?` plus the join logic adds `/` before non-** segments only when the prior segment didn't end with `?`. For a 3-segment `**/auth/**`: parts=[`**`,`auth`,`**`] ‚Üí out=[`(?:.*/)?`, `auth`, `/.*`]. Joined: `(?:.*/)?auth/.*`. So `^(?:.*/)?auth/.*$`. Test "src/auth/login.ts" matches because `(?:.*/)?` consumes `src/`. Good. But pattern `foo/**/bar` ‚Üí parts=[`foo`,`**`,`bar`] ‚Üí out=[`foo`, `/(?:.*/)?`, `/`, `bar`] (wait ‚Äî `i>0 and not out[-1].endswith('?')` ‚Äî `/(?:.*/)?` ENDS in `?`, so NO `/` is added before `bar`). Result: `^foo/(?:.*/)?bar$`. Test "foo/x/y/bar" ‚Üí matches. Test "foo/bar" ‚Üí `(?:.*/)?` is empty, gives `foo/bar` ‚Üí matches. Good. But "foo/x/bar/extra" ‚Üí no match. Good. Edge case: `foo/**` ‚Üí parts=[`foo`,`**`] ‚Üí out=[`foo`, `/.*`] ‚Üí `^foo/.*$`. "foo/" matches (empty `.*`). "foo" alone does NOT match (no trailing slash). Hard-gate `_deprecated/**` will NOT match the bare directory `_deprecated` if someone passes that directory as a path. Self-test doesn't cover bare-dir.
**Impact:** Edge cases mostly correct but undocumented and untested. A find / git-status output passes file paths not directories, so probably benign for hard-gates today. Risk for future use of glob_to_regex against directory inputs.
**Fix:** Add self-test cases for: bare-directory (`foo/**` against `foo`), middle-** with multiple segments (`a/**/b/**/c`), pattern with leading `**` plus suffix (`**/x` against `x` ‚Äî does it match? Depends on `(?:.*/)?` consuming nothing).

### [medium] HUMAN_GATE_QUEUE python3 heredoc has shell-quoting fragility
**Source:** redteam
**Evidence:** god-review.md:953-968 ‚Äî env vars are passed FINDING_ID, FINDING_FILE, FINDING_LINE_RANGE, FINDING_DESC, FINDING_DIFF ‚Äî but the python3 invocation uses `'...'` single-quoted body that contains backslash-escaped quotes inside f-strings: `os.environ[\"FINDING_ID\"]`. The outer `python3 -c '...'` single-quote is the bash shell's single-quote ‚Äî so backslashes inside are literal. Python parses `os.environ[\"FINDING_ID\"]` as `os.environ["FINDING_ID"]` correctly. But if FINDING_DIFF contains a triple-backtick (very common ‚Äî Architect outputs are diffs!), the f-string `f"\n```\n{os.environ[\"FINDING_DIFF\"]}\n```\n"` produces text with nested triple-backticks that break markdown rendering downstream.
**Impact:** Cosmetic for markdown rendering, but the queue entries become unreadable. More serious: if FINDING_DIFF contains the actual heredoc marker text, the python3 invocation itself doesn't break (it's a Python string, not a bash heredoc), but the resulting report.md sections become malformed. Combined with NO escaping of triple-backticks, the report.md may have broken section boundaries that downstream parsers (e.g., the orchestrator's own re-parse in next round's 3a) misread.
**Fix:** Wrap FINDING_DIFF in 4-backtick fence instead of 3-backtick: `f"\n````\n{...}\n````\n"`. Or escape the diff via base64 + indicate it's encoded.

### [low] codex-invoke.sh shared-home spinlock has a race between stale-detection and lock-acquisition
**Source:** redteam
**Evidence:** codex-invoke.sh:100-113 ‚Äî checks `[ -d "$LOCKDIR" ]`, computes age, removes if stale, THEN starts `while ! mkdir "$LOCKDIR"`. If two processes detect stale at the same moment, both rmdir, both then mkdir successfully (since dir is gone) ‚Äî but only one mkdir wins. Other gets EEXIST and retries. So it's not catastrophic, but the rmdir of a not-yet-stale lock by a racing process is possible: process A sees age=599, doesn't rmdir; process B sees age=601 (one second later), rmdirs while A is mid-operation. A's trap-EXIT then rmdir's a lock B is holding.
**Impact:** Rare. Concurrent codex-invoke calls during shared-home fallback (no CODEX_HOME_1/2 set) can step on each other. State unlikely-corrupted but ephemeral session.json could be lost.
**Fix:** Use mtime-difference + atomic rename pattern, or make the LOCK_TIMEOUT default longer than any reasonable single Codex run (e.g., 1800s = 30min). Or just require flock and abort if unavailable.

### [low] check_phase_drift uses `awk '/^## Phase 3/{p=0} p; /^## Phase 0/{p=1}'` ordering bug
**Source:** redteam
**Evidence:** env-helpers.sh:333 ‚Äî awk pattern is order-sensitive. In awk, action-by-pattern executes in source order. The pattern `{p=0} p; /^## Phase 0/{p=1}` means: when "## Phase 3" matches, set p=0; then on every line print if p=1; then on "## Phase 0" set p=1. Line "## Phase 0" itself: hits Phase-0 pattern, sets p=1, but the `p;` already fired with the OLD p value before the assignment ‚Äî so the "## Phase 0" header line is NOT printed. The Phase-3 line IS printed (p=0 hits before its assignment also? Wait: order is `/Phase 3/{p=0} p; /Phase 0/{p=1}`. For line "## Phase 3": pattern Phase-3 fires p=0; then `p;` evaluates with new p (now 0) ‚Üí not printed. So Phase-3 boundary line itself is excluded. Mostly OK, but awk's evaluation rule is statement-by-statement so the assignment IS visible to the print. Still, the `db` extraction at line 334 is `awk '/^## Phase 0/{p=1} p' "$b"` ‚Äî reads from Phase 0 to end-of-file. If god-report.md happens to contain Phase 3 headers (it doesn't today, but could in future), the diff is asymmetric: god-review.md cuts at Phase 3, god-report.md keeps everything.
**Impact:** Drift detection asymmetric. False-positive drift warnings if god-report.md ever gains a Phase 3 section.
**Fix:** Use the same start/end pattern in both extracts, or use a single awk with `/^## Phase [012]/,/^## Phase 3/` range pattern.
