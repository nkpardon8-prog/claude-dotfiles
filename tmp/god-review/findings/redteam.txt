[redteam] LOAD_BEARING_FRAGILITY: The "real Phase 3 round loop" is split across separate
                                  ```bash``` fences and cannot execute as a loop.
                                  — god-review.md:779-813 (while opens) → 1291-1408 (done closes)
Confidence: definite
Evidence:
  Line 779-813 contains the block:
      ```bash
      ...
      while [ "$LOOP" = "true" ] || [ "$ROUND" -le "${MAX_ROUNDS:-5}" ]; do
        echo "=== Round $ROUND ==="
        ...
      ```
  The matching `done` is at line 1406, inside a *different* fenced bash block
  (fence range 1291-1408). Between them, eight independent ```bash``` fences exist
  (832-845, 852-866, 907-969, 974-981, 989-1055, 1059-1125, 1129-1258, 1262-1287) —
  each is a separate bash invocation by the agent harness.
  Bash will error at the end of the first block with `syntax error: unexpected end of file`
  because `do` was never closed. The intermediate blocks then run as their own one-shot
  shells. The `done` at 1406 also runs in isolation and errors with `done: command not found`
  or `syntax error near unexpected token done`.
Attack/Failure Scenario:
  - Bounded mode never actually loops. ROUND is always 1 — set in the failed first block,
    persisted via write_env, but never incremented because the increment lives inside the
    "loop body" that is never inside a real loop.
  - --loop mode is even worse: the convergence check (3 consecutive clean rounds) never
    fires because CONSECUTIVE_CLEAN_ROUNDS only ever increments inside the unreachable
    "loop body" path of block 1291-1408.
  - All exit codes 2-7 wired in 2dd64ec are reachable only via the never-running termination
    block.
  - Commit 93812eb claims "real Phase 3 round loop — while construct with bounded + indefinite
    branches". That commit message is wrong. The construct is syntactically a loop, but at
    runtime the loop body never executes the post-fix branching.
Fix: Either collapse Phase 3 into ONE bash fence (keeping the python heredocs as files invoked
  externally), or replace the markdown-narrative orchestration with a single dispatcher script
  in lib/ that the agent invokes once per round. The current "narrative + multiple fences"
  pattern fundamentally cannot host a shell loop.

[redteam] INVARIANT_NOT_ENFORCED: glob_to_regex produces wrong regex for nearly every
                                  hard-gate pattern. is_hard_gate() silently approves
                                  files it must reject.
                                  — lib/env-helpers.sh:32-60 + hard-gates.txt
Confidence: definite
Evidence:
  Live test of the existing glob_to_regex() against hard-gates.txt patterns:
      GLOB=migrations/**         REGEX=^migrations/(?:.*/)?$
      GLOB=tests/**              REGEX=^tests/(?:.*/)?$
      GLOB=**/auth/**            REGEX=^(?:.*/)?auth/(?:.*/)?$
      GLOB=*.test.*              REGEX=^[^/]*\.test\.[^/]*$
      GLOB=package.json          REGEX=^package\.json$
  Live match results:
      migrations/001_init.sql     vs  migrations/**         → MISS (regex requires trailing /)
      tests/foo.spec.ts           vs  tests/**              → MISS
      src/auth/login.ts           vs  **/auth/**            → MISS
      src/foo.test.ts             vs  *.test.*              → MISS (regex anchors `^[^/]*` only matches no-slash)
      frontend/package.json       vs  package.json          → MISS
      .env.local                  vs  .env.*                → MATCH (this is the only one that works)
  Two algorithm bugs:
    1. `**` followed by `/` is consumed and emits `(?:.*/)?` — but **the `/` after `**` is
       then ALSO consumed** in the i+=1 fall-through, and the rest of the pattern after
       `tests/**` is empty. So `tests/**` produces `^tests/(?:.*/)?$`, which does NOT match
       `tests/foo.spec.ts`. The regex needs `.*` after the `(?:.*/)?` to match the basename.
    2. `^[^/]*\.test\.[^/]*$` cannot match `src/foo.test.ts` because of the path-anchor.
       Hard-gate intent is "any *.test.* in any directory" — pattern needs a leading
       `(?:.*/)?` prefix, OR the matcher should run on basename only, OR globs need a
       leading `**/` recursive wildcard auto-applied.
    3. `package.json` matches only the top-level file, not `frontend/package.json` —
        the hard-gates.txt does not anchor `**/package.json`. Either the patterns must
        all start with `**/`, or glob_to_regex must auto-recurse single-segment patterns.
Attack/Failure Scenario:
  - The Architect produces a fix that targets `migrations/2026_drop_users.sql`. is_hard_gate
    returns 1 (not gated). The hard-gate check at line 965 prints nothing, the Editor
    applies the change, gates pass (cargo/npm don't typecheck SQL), and the migration is
    committed and pushed. The README/CRITERIA.md promise that schema migrations are NEVER
    auto-modified is silently violated.
  - Same path for `*.test.*`, `tests/**`, `**/auth/**`, monorepo `frontend/package.json`,
    nested `db/migrate/2026_users.rb`. All five entire categories of hard-gates are bypassed.
  - The earlier filename-prefix injection guard (case pattern at line 953-957) does NOT
    catch any of these — it only blocks shell metacharacters, not category violations.
  - The fallback defense in lib/editor-agent.md (line 47-58) is a documented prompt to the
    sub-agent — that is "safety by documentation", not enforcement. A tired or misinstructed
    Editor agent will gladly apply the change.
Fix: Replace glob_to_regex with `python3 -c "import fnmatch,sys; sys.exit(0 if fnmatch.fnmatch(sys.argv[1], sys.argv[2]) else 1)"`,
  OR rewrite glob_to_regex to (a) emit `.*` after `**/` not just `(?:.*/)?`, (b) auto-prefix
  patterns lacking a leading `/` or `**` with `(?:.*/)?`, (c) handle basename-only patterns.
  Then ADD a unit test in lib/e2e-test.sh that runs every pattern in hard-gates.txt against
  a known-positive and known-negative path. Any future change must keep that test green.
