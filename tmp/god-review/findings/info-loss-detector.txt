**Scope:** ~/.claude-dotfiles/commands/god-review/ + god-review.md

The cross-block-var-persistence FIX is partially working (env-helpers.sh exists; sourced from 23 of 24 bash blocks) BUT the write_env whitelist is incomplete — it persists stack signals and CLI flags but NOT the loop counters, pre-scan results, or tunable constants that downstream fences depend on. ### 1. [CRITICAL / definite] Tunable Constants block has no persistence at all
**Location:** god-review.md:161-172 (block) + lib/env-helpers.sh:20-24 (whitelist)
**Set in:** block at line 161 (SHRINKAGE_PCT, PERF_REGRESS_PCT, INSTABILITY_RATE, FROZEN_UNITS_CAP, SECRET_LEN_FLOOR, TEST_FILE_LINE_FLOOR, LATE_IMPORT_LINE, SPINLOCK_TIMEOUT_SEC)
**Read in:** Phase 1 pre-scan block (line 314: `$SECRET_LEN_FLOOR`), Phase 3 churn block (line 1080: `$SHRINKAGE_PCT`, `$TEST_FILE_LINE_FLOOR`), Phase 3 perf block (line 1163: `$PERF_REGRESS_PCT`), termination check (line 1337: `$INSTABILITY_RATE`, line 1363: `$FROZEN_UNITS_CAP`).
**Gap:** (a) the Tunable-Constants block does NOT source env-helpers.sh (only block in file lacking the pattern). (b) None of these eight vars are in the write_env whitelist in lib/env-helpers.sh:20-24, so even adding the source line would not persist them. Every downstream comparison silently sees an empty value — `[ "$FROZEN_UNITS_COUNT" -gt "" ]` triggers a bash "integer expression expected" error, and `awk -v shrink="" '$2/($2+$1) > shrink'` divides by/compares against unset. The pre-scan secret length floor falls back to "" which `[ "${#val}" -lt "" ]` likewise breaks.
**Fix:** add `WORKDIR=...; source env-helpers.sh` at block start, add a `write_env` at end, AND extend the whitelist to include all eight tunables.

### 5. [HIGH / likely] state.json full-rewrite at Phase 1b destroys prior-run fields on non-resume
**Location:** god-review.md:383-413
**Write pattern:** heredoc → sed → mv (atomic write to disk, but FULL replacement of contents)
**Gap:** When invoked WITHOUT --resume on a tree where tmp/god-review/state.json already exists (e.g., user ran a fix loop, killed it, started again without --resume), Phase 1b unconditionally writes the fixed-schema heredoc — destroying any prior `human_gate_emitted`, `kept_fixes`, `reverted_fixes`, `frozen_units`, `churn_ledger`, `frozen_added_per_round`, `architect_malformed_per_round` entries. The atomic-write mechanism is correct (.tmp → mv), but the operation is replace-not-merge, so atomicity protects against SIGKILL but not against logical data loss across invocations.
**Fix:** wrap the heredoc init in `if [ ! -f tmp/god-review/state.json ]; then ... fi`, OR read + json-merge with python instead of overwriting.
