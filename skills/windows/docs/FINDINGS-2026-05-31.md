## The two `/macmini` conflicts (re-check at runtime, with fallback)

`/macmini` is the architectural ancestor and its SKILL encodes two findings that
**contradict what was observed on this Windows session**. Both worked here — but
re-check at runtime and keep the fallback ready. Do not let macmini's docs talk
you out of the verified-here behavior, and do not assume the Windows behavior
holds if the runtime probe says otherwise.

### Conflict 1 — `click_at` deprecation

- **macmini says:** synthetic CDP clicks (`mcp.click_at` / `evaluate_script`
  `el.click()`) are rejected by the CRD canvas (`isTrusted=false`); macmini
  deprecated `click_at` and routes clicks through `cliclick` on the mini via gist
  transport.
- **On this Windows session:** direct `mcp.click_at({x,y})` **reached the Windows
  host** and landed accurately three times (checkbox toggle, field focus). No
  on-host agent was needed.
- **Why the clash is genuinely unresolved:** either macmini's `isTrusted` claim
  was over-cautious / specific to an older CRD or Mac-host quirk, or something
  differs between the Mac and Windows CRD sessions. Not settled.
- **Skill behavior:** build `/windows` on direct `click_at` (verified). **Re-check
  every first session** with the toggle-and-confirm smoke test (toggle a harmless
  checkbox, screenshot-confirm). **Fallback** if it ever stops landing: surface
  to the user; there is no on-host agent for `/windows`, so a broken click
  channel is a STOP-and-escalate, not a silent reroute.

### Conflict 2 — CRD a11y tree `ignored`

- **macmini says:** CRD's a11y tree returns `ignored` for nearly every control;
  `mcp.click({uid})` cannot reach the side-panel toggles, so the user clicks
  "Synchronize clipboard" / "Send system keys" manually once.
- **On this Windows session:** `take_snapshot` **exposed clickable uids** for CRD
  controls — closed a dialog and the options panel by uid (LAYER-1 worked).
- **Skill behavior:** drive LAYER-1 by uid-label by default, but **probe at
  runtime** — if `take_snapshot` returns `ignored` for the CRD controls, fall
  back to coordinate `click_at` on the panel control's on-screen location, or ask
  the user to perform the toggle manually. The "CRD a11y" smoke test learns which
  mode this session is in. Match controls by **label** each snapshot; never
  hardcode a uid (this session saw `2_7`/`2_8`/`2_43`/`2_44`/`2_62` — all dead on
  the next snapshot).

## Text channel — why clipboard is v2, not v1

The brief's original plan ("enable Synchronize clipboard, then `pbcopy -> Ctrl+V`")
was obsoleted twice:

1. Even with "Synchronize clipboard" **checked**, `pbcopy -> Ctrl+V` pasted stale
   *Windows* clipboard content, not the Mac string. **Root cause:** Chrome's
   clipboard read for CRD needs real **OS-foreground focus / a user gesture**,
   which a CDP-driven `bringToFront` cannot supply — that root cause, not the
   toggle, is why clipboard is v2.
2. The shift-map breakthrough (Addendum F) made it moot for v1: capitals/symbols
   are typable via `press_key("Shift+<base>")`, pure CDP, zero external deps,
   zero credential-leak surface. So **v1 types "whatever it wants" in ASCII** with
   no clipboard, no gist, no CRD on-screen-input. Unicode / emoji / bulk paste
   stay v2 (clipboard or file channel).

## Coordinate model (the math, restated)

- `devicePixelRatio = 2`, host streams `1920x1080` into a `<canvas>`.
- `getBoundingClientRect()` and `click_at` are **both CSS px** -> rect-derived
  click coords are **NOT** /DPR.
- Only a target eyeballed off a raw screenshot is /DPR (screenshots save at
  `innerW x DPR`, e.g. 3420x1970), and that path must also subtract the canvas
  letterbox offset (`rect.x`/`rect.y`).
- Read DPR / host size live via `evaluate_script`; never hardcode `2`, `1920`, or
  `1080` into the click formula — read `hostW`/`hostH` from the helper. The
  session ID (`f9a451b1-...` this session) is ephemeral — never hardcode it; bind
  by title.

## Out of scope (v1) / v2

- Unicode / emoji / bulk paste — clipboard or file channel (v2). Clipboard sync
  toggle being ON does not bridge under CDP (focus root cause above).
- Mouse-wheel scroll — no tool exists; use PageDown/Arrow or scrollbar-arrow
  clicks.
- Fullscreen-mode system shortcuts.
